# CM-1Side-Vision

## ? Git Branch Strategy

협업 시 코드 충돌 방지 및 릴리즈 관리 효율화를 위해 다음과 같은 Git 브랜치 전략을 사용합니다.

---

## 1. 개발 브랜치 관리

- 모든 개발자는 **개별 브랜치**를 만들어 작업합니다.
  - 브랜치 이름 예시: `feature/홍길동-기능명`, `bugfix/이몽룡-버그설명`
- 작업 완료 시, 테스트 후 **릴리즈 대기 브랜치로 병합**합니다.

---

## 2. 릴리즈 준비 브랜치

릴리즈가 필요할 경우, 아래와 같은 형식으로 릴리즈 대기 브랜치를 생성합니다:

```
Release/TEMP/AKC-1
Release/TEMP/제품명-버전번호
```

이 브랜치에서 QA 및 검증을 진행합니다.

---

## 3. 릴리즈 대기 중 수정

- 릴리즈 대기 브랜치에서 **이슈가 발생한 경우**, 해당 브랜치에서 직접 수정합니다.
- 별도 브랜치를 만들지 않고 바로 반영하여 QA를 지속합니다.

---

## 4. 릴리즈 확정

문제가 없고 릴리즈가 확정되면, 대기 브랜치를 **공식 릴리즈 브랜치로 전환**합니다:

```
Release-제품명
```

---

## 5. 공통 코드 배포

특정 제품과 무관한 **공통 기능 수정 또는 개선 사항**은 `master` 브랜치로 직접 반영됩니다.

---

## 6. 핫픽스 처리

릴리즈 후 문제가 발생할 경우, 아래 형식으로 **핫픽스 브랜치**를 생성합니다:

```
Hotfix/릴리즈브랜치이름/핫픽스설명
예시: Hotfix/Release-AKC-1/LoginCrashFix
```

빠르게 수정 후 원래 릴리즈 브랜치 및 `master`에 병합하고, 관련자에게 공유합니다.

---

## 7. 커밋 메시지 규칙

일관된 커밋 로그 관리를 위해 다음과 같은 커밋 메시지 규칙을 사용합니다:

### 기본 형식
```
[타입] 간단한 요약
```

### 타입 종류
- **Feature**: 기능 추가
- **Bugfix**: 버그 수정
- **Hotfix**: 긴급 수정
- **Refactor**: 리팩토링
- **Docs**: 문서 관련 변경
- **Test**: 테스트 코드 추가/수정
- **Chore**: 기타 잡다한 변경

### 커밋 메시지 예시
```
[Feature] 로그인 기능 추가
[Bugfix] 로그인 실패 시 오류 메시지 수정
[Hotfix] AKC-1 앱 실행 시 크래시 수정
```

---

## 8. 브랜치 네이밍 규칙

| 브랜치 종류 | 형식 예시 | 설명 |
|-------------|-----------|------|
| 기능 개발 | `feature/홍길동-기능명` | 각자 기능 개발 브랜치 |
| 버그 수정 | `bugfix/이몽룡-버그설명` | 버그 수정 전용 브랜치 |
| 릴리즈 대기 | `Release/TEMP/제품명-버전` | QA 전용 릴리즈 후보 |
| 릴리즈 확정 | `Release-제품명` | 배포 확정 릴리즈 브랜치 |
| 핫픽스 | `Hotfix/Release-제품명/수정내용` | 릴리즈 후 긴급 수정 |
| 공통 반영 | `master` | 공통 코드 수정 시 사용 |

---

## ? 워크플로우 흐름도

```
개발자 브랜치
      ↓
Release/TEMP/제품명-버전
      ↓
   QA 검증
  ↙      ↘
이슈 발견    검증 완료
  ↓           ↓
브랜치 내    Release-제품명
직접 수정       ↓
  ↓        배포 완료
  ↑           ↓
  └─────   문제 발생?
              ↓
         Hotfix/Release-제품명/설명
              ↓
           병합 및 공유
```

**상세 흐름:**
1. **개발 단계**: 각 개발자가 개별 브랜치에서 작업
2. **릴리즈 준비**: `Release/TEMP/제품명-버전` 브랜치로 병합
3. **QA 검증**: 테스트 및 품질 검증 실시
4. **이슈 처리**: 문제 발견 시 해당 브랜치에서 직접 수정
5. **릴리즈 확정**: `Release-제품명` 브랜치로 전환 후 배포
6. **핫픽스**: 배포 후 문제 발생 시 긴급 수정 브랜치 생성

---

## ? 시작하기

1. 개발 브랜치 생성: `git checkout -b feature/이름-기능명`
2. 작업 완료 후 릴리즈 대기 브랜치로 병합
3. QA 진행 및 이슈 수정
4. 릴리즈 확정 후 배포
